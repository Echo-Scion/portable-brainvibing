## Chapter 5: Directory Structure of `.orion/`

```
.orion/
├── _manifest.json        # Metadata of the entire index (36KB, auto-generated)
├── index.md              # Human-readable index map of all assets (135 entries)
├── log.md                # Episodic log across sessions
├── orion.db              # Main SQLite database — the core brain
├── episodic/             # Past session summaries (episodic memory)
├── matrix/               # Compressed YAML matrix of rules
├── sources/              # 118 files (.yml/.md) — processed copies of all assets
└── working/
    └── context.json      # Omni-Buffer (real-time IDE state)
```

### 5.1. `orion.db` — Main SQLite Database
Uses the virtual table **FTS5** (Full Text Search 5) for instant lexical search:
```sql
CREATE VIRTUAL TABLE memories USING fts5(
    id UNINDEXED,
    title,
    content,
    tags
);
```
Every `.agents` file is hashed with SHA-256 before being upserted, preventing duplication. Instant search query without cloud costs:
```sql
SELECT * FROM memories WHERE memories MATCH 'auth OR jwt OR session';
```

### 5.2. `matrix/` — YAML Rule Matrix
The YAML version of each rule file. Generated by `compile_rules.py`. Its purpose: so that NanoBrain and Python scripts can parse rules programmatically without parsing Markdown manually.

### 5.4. `index.md` — Human Navigation Map
A 135-line file listing every asset in the ecosystem, grouped by layer:
- **Infrastructure Layer**: Canons, rules, skills, workflows.
- **Context Layer**: Business strategy, architecture, roadmap files.
- **Project Docs Layer**: `LEARNINGS.md`.
- **Raw/Source Layer**: Raw Python scripts (harness, compress, etc.).

### 5.4. `_manifest.json` — Machine Metadata
A large JSON file storing hash metadata, last processed date, and classification type for each file. Used by `orion_ops.py` to determine if a file has changed since the last ingest.

## Chapter 6: Relational Triplet Graph (Preventing Compilation Bugs)

### 6.1. Core Concept
Source code is not just text; it has structural relationships between entities. The system extracts **Semantic Triplets** (Subject - Predicate - Object) from each file:
```json
{
  "subject": "UserController",
  "predicate": "depends_on",
  "object": "AuthService",
  "context": "UserController requires AuthService to validate JWT tokens on line 45."
}
```

### 6.2. Practical Impact
If the AI is asked to refactor `AuthService`, the Triplet Graph immediately issues a warning:
*"UserController depends on AuthService. You MUST update both."*
This eliminates compilation bugs due to blind refactoring.

### 6.3. Triplet Injection
The `knowledge-extraction.md` workflow instructs the AI to:
1. Read a specific source file.
2. Extract 3-5 Semantic Triplets.
3. Run `orion_ops inject_triplets` which executes SQL `INSERT` into the edge table.

## Chapter 7: Universal AST Parser (Code Summarizer)

### 7.1. The Problem
Feeding 2,000 lines of code into the AI's context is extremely expensive in terms of tokens. Most lines are comments, whitespace, and repetitive logic blocks that are irrelevant for architectural understanding.

### 7.2. The 4-Tier Parsing Engine
The file `scripts/utils/ast_parser.py` implements a multi-tiered architecture to process source code and generate a structural skeleton:
1. **Rust Token Killer (RTK)**: Tries to execute `rtk read --level aggressive` for instantaneous Rust-based hollow AST generation.
2. **Native Python `ast`**: Falls back to the built-in `ast` module for purely accurate Python parsing.
3. **C-Bindings via `tree-sitter`**: Falls back to `tree_sitter` for robust extraction of JS, TS, and Dart files.
4. **Regex Heuristics**: Acts as a graceful last-resort fallback for unsupported files or missing dependencies.

**Original Code (50 lines):**
```python
class AuthService:
    # This function logs the user in
    def login(self, user_id: str):
        print("Logging in...")
        db.connect()
        if not user:
            raise Exception("User not found")
        return True
```

**AST Footprint (2 lines):**
```python
class AuthService:
    def login(self, user_id: str) -> bool: pass
```

Reduction: **~90%** size, with perfect retention of class names, method names, and parameter types. The AI knows the architecture without reading the implementation.

## Chapter 8: True LightRAG Engine (Dual-Retrieval & NanoBrain)

### 8.1. The Hybrid RAG Architecture
The `.orion` Knowledge Base is not just a vector database or a simple keyword searcher. It implements a **Hybrid LightRAG (Graph + Vector + Lexical)** architecture, explicitly designed for maximum portability (Zero C-Extensions like `sqlite-vss`) and Graceful Degradation.
1. **Lexical Indexing (FTS5)**: Fast keyword matches using SQLite's native `fts5`.
2. **Vector Embeddings (nomic-embed-text)**: Taps into Ollama locally to convert texts into JSON vectors, stored directly in standard SQLite. Calculates Cosine Similarity in pure Python `math` in milliseconds.
3. **Unified Triplet Graph (SQLite)**: Mechanical edges (`imports`, `defines`) are extracted strictly locally via the 4-tier AST parser, whilst Deep Semantic Edges (Subject-Predicate-Object logic) are generated by Cloud LLMs via `inject_triplets` into the unified `orion.db` (eliminating the legacy JSON dual-graph sync issues).

### 8.2. Granular State Routing (NanoBrain)
To support massive vector intelligence on laptops with limited RAM, NanoBrain features **Granular State Control**. Users can selectively activate capabilities without loading heavy local generative LLMs:
- `brain nanobrain on`: **FULL Mode**. Generative LLM (via Ollama) and Vector Embeddings (`nomic-embed-text`) are both active. *Note: We do not force local "dumb" models for complex reasoning. Deep extraction is always delegated to Cloud.*
- `brain nanobrain embed_only`: **Vector-Only Mode (Recommended)**. Gives you world-class semantic search via `nomic-embed-text` without eating RAM for generative AI. This is the **Graceful Degradation** philosophy in action.
- `brain nanobrain llm_only`: Vector logic disabled, generative AI active.
- `brain nanobrain off`: Complete fallback to pure FTS5 Lexical Search.

### 8.3. Active Mode (`brain sync`)
When the user types "fix the database", the routing sequence is:
1. **Semantic Query Extraction**: NanoBrain generates an embedding for "database".
2. **Dual-Retrieval (Vector + FTS5)**: Runs a combined query scoring exact keyword matches against semantic vector distances.
3. **Dynamic Rule Filtering**: The best nodes are routed to the AI's context window.

This guarantees the expensive Cloud AI models only receive laser-focused context.

### 8.3. Compression Hook (`nano_compressor`)
For very heavy tasks, the NanoBrain can be utilized manually or via hooks to compress context.
1. `nano_compressor.py` is invoked to read markdown files.
2. It summarizes the text using the local LLM.
3. The compressed summary replaces the original content or is stored in `episodic/` as a permanent episodic memory.

## Chapter 8.5: The Unstructured Knowledge Base (`context/`) & Obsidian Integration

### 8.5.1. The Second Brain Concept
To support human-driven ideation alongside AI deterministic tasks, the system features a `context/` directory (The Second Brain). It is conceptually decoupled from `.agents` but heavily utilized by `.orion`.
- `context/raw/`: Drop-zone for raw, unstructured data (meeting transcripts, quick thoughts, copy-pasted logs).
- `context/wiki/`: Structured, tagged, and formatted Markdown knowledge base.

### 8.5.2. Raw-to-Wiki Pipeline
The AI does not rely on heavy LLMs for basic ingestion. A specialized worker script (`raw_ingest.py`) continuously (or on-demand) processes the `context/raw/` directory. It uses a 0-token frequency-based algorithm to extract dominant keywords, structures the text with YAML Frontmatter, and migrates the file into `context/wiki/` safely.

### 8.5.3. Native Obsidian Vault Support
The entire `.orion/` architecture and `context/` directory are physically mapped as a native Obsidian Vault. 
- The `linkify.py` script automatically scans the graph and injects `[[wikilinks]]` into the rule matrix and sources.
- This allows developers to open the project root in Obsidian, view the interactive Knowledge Graph, and seamlessly navigate between human `context/wiki/` notes and the AI's `.agents/` rules.

### 8.5.4. Proactive Synchronization (Git Hooks)
Instead of relying on the developer to manually run ingestion commands, the system integrates proactively with Git. A `.git/hooks/post-commit` script is automatically injected. Upon every commit, the hook runs `python .agents/scripts/orion.py brain sync` in the background, ensuring the AI's internal state remains perfectly aligned with the codebase's evolution.

## Chapter 8.6: Memory & Knowledge Evolution

The Brainvibing architecture is not a static database; it proactively prunes, synthesizes, and resolves conflicts within its own knowledge base.

### 6.1. Smart Memory Hygiene (Context Pruning)
When memory compresses, the system avoids pruning critical information using a mechanical eligibility filter.
- Every node in the `.orion.db` SQLite database tracks an `access_count`.
- When the AI performs a semantic search (GraphRAG), the system automatically increments the `access_count` for the recalled nodes.
- When `compress_memory.py` runs, it preserves highly-accessed nodes and nodes younger than a specific threshold, ensuring long-term memory remains relevant.

### 6.2. Contradiction Resolution Engine
AI agents frequently encounter contradicting instructions (e.g., File A says "Use Riverpod", File B says "Use Provider").
- During GraphRAG ingestion, `contradict.py` automatically scans for colliding predicates (e.g., `state_management` points to two different libraries).
- It logs these collisions into the `contradictions` SQL table.
- The AI is then prompted to resolve these contradictions algorithmically by adjusting a `confidence` score (0.0 to 1.0) on the edges, ensuring a single source of truth without deleting historical data.

### 6.3. Activity-Based Temporal Pulse
Instead of using rigid calendar-based summaries (Daily/Weekly/Monthly) which break when a project is paused, the `.agents` ecosystem uses an **Activity-Based** synthesis protocol (`consolidate.py`):
1. **Session Pulse**: Triggers when the user calls `/session-offload`. Summarizes the immediate working context.
2. **Milestone Synthesis**: Triggers after N sessions. Consolidates multiple sessions into a major architectural milestone.
3. **Epoch Consolidation**: Triggers after N milestones. Compresses deep history into macro-level evolutionary logs.

## Chapter 8.7: Execution Resilience (Crash-Proofing)

AI agents are prone to failure during multi-step executions if the IDE crashes or if context is lost.

### 7.1. Identity Layer Bootstrap
When an agent wakes up in a new environment, it lacks a foundational identity.
- `pre-agent-wake.py` detects if `context/AGENT_IDENTITY.md` is missing.
- If missing, it automatically auto-generates it based on a template, merging historical `LEARNINGS.md` into the identity profile, ensuring the AI never starts with a blank slate.

### 7.2. Formal State Machine (Crash-Proof Memory)
The architecture completely separates static environment data (`context.json`) from active execution data (`execution_state.json`).
- When executing a complex workflow, the AI logs phase transitions (`PLANNING`, `EXECUTING`, `VERIFYING`) and active tasks via the `orion.py fsm` engine.
- **The Crash Recovery Hook**: If the IDE crashes, `pre-agent-wake.py` detects the dangling execution state and automatically injects a `recovery_instruction` into the AI's Omni-Buffer, forcing the AI to resume exactly the task it was performing at the millisecond of the crash.

### 7.3. Sub-Agent Worker Delegation
To avoid token bloat during massive repository scans, the Architect AI delegates tasks using the `worker-delegate` skill. Sub-agents execute narrow tasks and write their results to `.agents/working/delegations/` files instead of polluting the main AI's context window.
